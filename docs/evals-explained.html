The Skill Eval Framework, in plain English

What the framework measures, why each measurement exists, and how to read a regression report.

What an eval is, here

An eval is a repeatable test that runs a real LLM against fixed prompts and scores the result. Unlike a unit test, the LLM's output varies run-to-run, so we run each prompt several times (called trials) and aggregate.

The output is a set of numbers that answer: is this skill worth shipping, and did this change make it better or worse?

1. Three tiers, three cost levels

Not all checks cost the same, so we layer them.

TierWhat it checksCostWhen
0 — Lint Static checks: does the skill's frontmatter parse, do cross-references point to real files, does the JSON schema validate. Free, ms. Every PR.
1 — Trigger Send ~20 prompts to Claude, watch whether the skill loads when it should (and stays quiet when it shouldn't). ~$0.05 / case Changed skills + nightly.
2 — Quality Send a few harder prompts twice — once with the skill available, once without — and grade both. ~$0.20 / case Changed skills + nightly.

2. Discovery: did the right prompt load the skill?

Skills in Claude Code are loaded lazily. Claude reads short skill descriptions in its system prompt and decides per-prompt whether to pull in each skill's full body. A bad description means the skill never loads when needed — or loads when it shouldn't.

The trigger eval ships ~10 prompts of each kind:

  • Positive probes — the skill should load.
  • Negative probes — the skill should not load, but the prompt is topically close enough to be a plausible trap (e.g. a JUEL/Camunda-7 prompt next to a FEEL skill).

The three numbers it computes

Precision

Of the times the skill loaded, how often was that the right call?

High precision = few false alarms.

Recall

Of the prompts that should have loaded the skill, how often did it?

High recall = description is pulling its weight.

F1 is a single combined number (the harmonic mean of the two). Handy as a headline, but bad for decisions because it hides which of the two is actually hurting.

A skill with precision 1.0 but recall 0.2 is over-cautious. One with precision 0.4 but recall 1.0 is over-eager. Both look mediocre on F1 alone — but the fixes are completely different.

3. Quality: when the skill loads, is the answer better?

The naive version — "run the prompt with the skill loaded and check if the answer is good" — tells you nothing. Claude might already know how to do the task from training. You'd be measuring Claude, not your skill.

The fix is paired arms: run the same prompt twice.

with_skill arm

All repo skills are exposed to Claude, including the one under test.

camunda-feel, camunda-bpmn, camunda-forms…

without_skill arm

All repo skills are exposed except the one under test. Siblings stay (matching what a real user with the plugin installed sees).

camunda-feel, camunda-bpmn, camunda-forms…

Each arm is graded. The number that matters is the difference:

skill_help = with_skill_pass_rate − without_skill_pass_rate

This is the marginal lift your skill provides. If both arms pass 90% of the time, the skill is doing nothing useful. If with_skill is 90% and without_skill is 30%, the skill is teaching Claude something genuinely load-bearing.

4. Skill flavors set expectations

Two flavors live in this repo, and they look very different on the dashboard.

FlavorWhat it coversTypical skill_help
Knowledge-additive
camunda-bpmn, camunda-c8ctl		 Proprietary surfaces Claude doesn't know from training. Without the skill, Claude invents wrong syntax. Big — 30–70pp.
Correctness-protective
camunda-feel		 Topics Claude mostly knows, but with subtle gotchas. The skill catches the rare trap. Small but real — 5–25pp.

Knowing which flavor you're shipping sets reviewer expectations. A 10pp skill_help on camunda-c8ctl would be alarming; the same number on camunda-feel is normal.

5. The judge and the verifiers

The judge

Each case has an expectations[] list — short prose checks like "Use if-then-else with an else branch". After the agent answers, a separate Claude instance (Sonnet, isolated, with only Read/Write tools) reads the trial transcript and the output file, then grades against the expectations. It writes a grading.json with a per-expectation verdict and an overall pass_rate.

A trial passes if pass_rate ≥ 0.5 AND all non-skipped verifiers pass.

Verifiers

The judge can be fooled by output that reads right but is behaviorally wrong. A FEEL expression can look fluent and still evaluate to the wrong number. A BPMN file can look reasonable and fail to parse.

Verifiers shell out to real tooling against the agent's output file:

  • feel-evaluate — runs c8 feel evaluate against the real engine, compares stdout to a fixed expected value.
  • bpmn-lint — runs c8 bpmn lint --quiet, fails on any parse error or lint violation.

Three outcomes: pass fail (blocks the trial) skipped (couldn't run — recorded but doesn't block).

6. The baseline — the line in the sand

When you're happy with a skill, you run a full eval (3 trials per case, ~$8.50 for camunda-feel) and promote the results into skills/<name>/evals/baseline.json. This file is checked into git. It pins:

  • The headline numbers (trigger F1, with/without pass-rates, skill_help).
  • The model used (so a baseline isn't compared across model changes by accident).
  • The trial count.
  • The regression thresholds.

Every subsequent PR re-runs evals and compares to this baseline. The output is rendered as a markdown delta (suitable for posting as a PR comment once that wiring lands). Promotion is deliberate — never automatic. Bumping a baseline lands in its own PR with a written rationale.

7. The asymmetric regression rule

This is the rule that decides whether a PR check fails or just warns. Let's unpack the jargon first.

The vocabulary

  • "pp" = percentage points, not percent. If a pass-rate was 80% and is now 73%, that's a 7pp drop. (A "7% drop" would be ambiguous — 7% relative to 80% is only 5.6pp. The "pp" unit avoids that confusion.)
  • "Regress at >5pp drop" means: if the metric drops more than 5 percentage points vs. the baseline, the PR check fails red and blocks merge. Drops of 5pp or less are within tolerance.
  • "Asymmetric" means the rule isn't the same for every metric. Some metrics block; some only warn.

The rule itself

MetricDrop thresholdSeverityWhy
with_skill_pass_rate >5pp regression The skill content got worse.
skill_help >5pp regression The skill helps less than before — threatens its reason to exist.
trigger precision >5pp regression Over-triggering grew — wasting context on every user.
trigger recall any warn only Under-triggering is fine if quality holds — Claude may just be answering unaided on prompts that don't need the skill.
trigger F1 any informational Derived from the two above — tracking it separately would double-count.
Why asymmetric? Because the two skill flavors look very different on the same dashboard. A camunda-feel-shaped skill might naturally have recall 0.30 — under-triggering by design, because Claude answers fine unaided most of the time — and that's not a bug. A blanket "any drop fails" rule would either be too strict for FEEL or too loose for c8ctl. The asymmetry lets one rule serve both.

8. The full flow on a PR

  1. You change a SKILL.md body or description.
  2. CI runs Tier 0 (fast), Tier 1 + Tier 2 on changed skills only.
  3. Each tier writes results into evals/<skill>/iteration-N/ plus a self-contained report.html.
  4. make compare diffs against baseline.json and renders a markdown table (PR-comment wiring is the natural next step).
  5. If with_skill_pass_rate, skill_help, or precision dropped >5pp, the check is red. Either fix it, or — if the drop is justified — re-promote the baseline downward in a separate PR with a written rationale.
  6. If green, the PR merges. To raise the bar after a genuine improvement, run a full 3-trial eval locally and make promote.

The point of all this: turn "is this skill change good?" from a vibe in code review into a number with a threshold.
